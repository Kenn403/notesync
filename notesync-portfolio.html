01 — The Problem

Students can't control the
quality of their own notes

Colleges hire volunteer note-takers to serve students with disabilities. But the system is fragile — note-takers are inconsistent, late, or absent. Students receive their notes hours after class, too late to ask questions while the material is fresh.

⏱️
Delayed delivery
Notes arrive hours or days after class, breaking the learning loop when retention and questions are most valuable.
🎲
Inconsistent quality
Every note-taker has different writing habits, speed, and comprehension. Students have no way to verify what was missed.
🔗
No context anchoring
Notes are disconnected from the slide content. Students can't trace "which slide was this about?" when reviewing.
✦
The insight
The job-to-be-done isn't "get notes." It's "understand what happened in class, on my own terms, at my own pace."
02 — User Stories

Designed around
real student needs

Primary persona: a university student with ADHD or a hearing impairment, enrolled in lecture-based courses, who currently relies on accessibility services for note support.

  • US-01 As a student with ADHD, I want live transcription during class so that I can focus on listening without worrying about missing content. P1
  • US-02 As a student with a hearing impairment, I want a synchronized text stream tied to each slide so that I can follow along in real time. P1
  • US-03 As a student reviewing before an exam, I want AI-generated summaries per slide so that I can quickly identify key concepts without re-reading the full transcript. P1
  • US-04 As a student uploading lecture slides, I want them auto-matched to transcript timestamps so that every piece of content has spatial context. P2
  • US-05 As a student studying independently, I want auto-generated flashcards from lecture content so that I can prepare for exams without building my own study materials from scratch. P2
  • US-06 As a student with slow processing speed, I want an AI chat interface I can ask questions about any lecture moment so that I can clarify without having to re-attend office hours. P3
  • US-07 As a student in multiple courses, I want a dashboard of all my recordings organized by course and date so that I can navigate my lecture history easily. P3

04 — Technology Stack

Every framework
chosen deliberately

No technology was chosen by default. Each decision has a reason tied directly to user needs, performance constraints, or accessibility requirements.

Layer Technology Why This One Alternatives Considered
Frontend Framework React 18 Component-based architecture maps perfectly to the slide panel + transcript panel + tab system. React's concurrent rendering handles live transcript updates without blocking the UI. Largest ecosystem for accessibility tooling (react-aria). Vue 3 (smaller ecosystem), Svelte (less hiring pool), Next.js (overkill for client-side app)
Styling Tailwind CSS Rapid iteration on design system with consistent spacing and color tokens. Dark mode support out of the box. Utility classes eliminate specificity wars in a component-heavy app. CSS Modules (verbose), Styled Components (runtime overhead), plain CSS (no system)
Live Transcription Deepgram API Sub-300ms latency via WebSocket streaming — critical for live class use. Superior accuracy for academic vocabulary. Built-in speaker diarization to separate professor from student questions. ~$0.0059/min cost is viable for student usage. OpenAI Whisper (no streaming, batch only), AssemblyAI (higher latency), Web Speech API (no offline, browser-only)
AI Summarization Claude API claude-sonnet-4 Best-in-class performance on structured educational content. Controllable output format (JSON) for summary + flashcard generation. Large context window handles full lecture transcripts. Anthropic's safety focus aligns with student-facing product. GPT-4o (comparable but pricier at scale), Gemini (weaker structured output), on-device models (quality tradeoff)
PDF Processing PDF.js pdf-lib PDF.js renders slide thumbnails directly in browser — no server round trip for upload preview. pdf-lib handles text extraction and page metadata for slide-to-timestamp anchoring. Server-side pdfminer (latency), iFrame embed (no programmatic access), PPT.js (PPTX only)
Backend / API Node.js Express JavaScript across the stack reduces context switching. Event-loop model handles concurrent WebSocket connections for multi-user transcription sessions. Fast to prototype with vast middleware ecosystem. Python/FastAPI (good but separate language), Go (performance overkill for MVP), Serverless (WebSocket complexity)
Real-time Socket.io Bidirectional streaming for live transcript push to client. Automatic reconnection on network drops (critical during class). Room-based architecture enables future multi-device sync. Native WebSockets (manual reconnect logic), Server-Sent Events (unidirectional), polling (unacceptable latency)
Database PostgreSQL Prisma Relational model fits the data perfectly: Users → Courses → Sessions → Slides → Transcript Segments. Prisma ORM provides type-safe queries. PostgreSQL's full-text search enables lecture search without Elasticsearch. MongoDB (less structure, no joins), Firebase (vendor lock-in), SQLite (not production-scalable)
File Storage AWS S3 Lecture audio files (avg 50–200MB) need durable cloud storage with signed URL access. S3 lifecycle policies auto-archive old sessions. Presigned URLs let clients upload directly — bypassing server bandwidth costs. Cloudinary (media focus, extra cost), local filesystem (not scalable), GCS (comparable, weaker US ecosystem)
Auth Auth0 University SSO integration (SAML/OIDC) is table-stakes for institutional adoption. Auth0 handles SSO, MFA, and RBAC out of the box. Reduces FERPA compliance risk by offloading auth to a SOC2-certified provider. NextAuth (React-only), Firebase Auth (no SAML), custom JWT (security risk, maintenance burden)
Deployment Vercel Railway Vercel for the React frontend — instant deploys, edge CDN, zero config. Railway for the Node.js backend — WebSocket support (Vercel doesn't support persistent connections), Postgres addon, environment management. AWS (too complex for MVP), Heroku (deprecated free tier), DigitalOcean (manual scaling)

// POST /api/sessions/:id/summarize // Called once after recording ends (or on-demand in review) const generateSlideSummaries = async (sessionId) => { const session = await db.session.findUnique({ where: { id: sessionId }, include: { slides: { include: { transcriptSegments: true } } } }); const summaries = await Promise.all(session.slides.map(async (slide) => { const transcript = slide.transcriptSegments .map(s => s.text).join(' '); const response = await anthropic.messages.create({ model: 'claude-sonnet-4-20250514', max_tokens: 1000, system: "You are an expert academic note-taker. 06 — Key Design Decisions

Tradeoffs made
with intent

Transcription Engine
✓ Deepgram (streaming)
OpenAI Whisper (batch)
Whisper has higher accuracy on clean audio but requires a complete audio file to process. For live class use, streaming with slight accuracy tradeoff is always the right call. Students need the transcript while the lecture is happening, not 30 minutes after.
Slide Sync Method
✓ Manual student tap
Automatic screen detection
Automatic slide detection via computer vision on a screen share is technically elegant but adds complexity and fails in rooms where the student can't see the projector clearly. Manual tap is 100% reliable and puts control in the student's hands — which is the whole product thesis.
AI Summary Timing
✓ Post-session batch
Live in-class AI summaries
Generating summaries during class feels fast but adds API cost and latency to the most critical flow (live transcription). Students don't read summaries in class — they read them after. Batching after the session is cheaper, faster, and doesn't compete with real-time transcription.
Data Model
✓ PostgreSQL (relational)
MongoDB (document)
The core data has clear relationships: one session has many slides, each slide has many transcript segments. Relational joins make full-text search across a session trivial. A document store would require application-level joins and lose ACID guarantees on concurrent writes during live sessions.
07 — Product Roadmap

Three phases to
institutional adoption

The roadmap is ordered by student impact, not technical complexity. Phase 1 delivers standalone value. Phase 2 adds institutional hooks. Phase 3 makes the product defensible.

Phase 1 — MVP (Weeks 1–6)
The Core Loop
  • Mic recording with live transcription
  • PDF/PPT slide upload + rendering
  • Manual slide-to-timestamp sync
  • Per-slide AI summary (post-session)
  • AI-generated flashcards
  • Session library / dashboard
  • PDF export of study notes
Phase 2 — Growth (Weeks 7–14)
Institutional Integration
  • University SSO (Auth0 SAML)
  • Canvas/Blackboard LMS integration
  • Auto slide import from course portal
  • Accessibility Services admin portal
  • AI chat: "Ask about this lecture"
  • Multi-device sync (phone + laptop)
  • Spaced repetition flashcard queue
Phase 3 — Scale (Month 4+)
Network Effects
  • Shared lecture libraries by course
  • Professor slide upload integration
  • Automatic slide detection (CV)
  • FERPA-compliant data controls
  • Analytics for Accessibility Services
  • Native iOS/Android app
  • Offline mode (on-device Whisper)
